## 4. 동작 방식

### 4-1. 인증 흐름
```
1. 브라우저 → POST /api/login (username/password)
2. 서버 → JWT 토큰 발급 (7일 유효)
3. 브라우저 → WebSocket 연결 + JWT 전송
4. 서버 → 인증 확인 후 세션 연결
```

### 4-2. 채팅 흐름
```
유저 메시지 입력
  → WebSocket으로 서버 전송
  → DB에 user 메시지 저장
  → 에이전트 순차 호출 시작:

    1. 이지두 (Gemini Flash)
       - 시스템 프롬프트 + 대화 히스토리 전달
       - 스트리밍 응답 → WebSocket으로 실시간 전송
       - 응답 DB 저장 + 히스토리 추가

    2. 김주희 과장 (GPT-4o)
       - 이지두 응답 포함한 전체 히스토리 전달
       - 스트리밍 응답 → 실시간 전송
       - 응답 DB 저장 + 히스토리 추가

    3. 민이 (Claude Sonnet/Haiku)
       - 이지두 + 김주희 응답 포함한 전체 히스토리 전달
       - 스트리밍 응답 → 실시간 전송
       - 최종 정리 (결론/액션/담당) 형식

  → turn_complete 신호 → 입력창 활성화
```

### 4-3. 컨텍스트 공유 핵심
각 에이전트는 이전 에이전트의 응답을 `[에이전트명]: 내용` 형식으로 assistant 메시지에 포함하여 전달받습니다. 이것이 봇 간 "대화"를 가능하게 하는 핵심 메커니즘입니다.

### 4-4. 개별 호출
메시지에 `@이지두`, `@김주희`, `@민이` 포함 시 해당 에이전트만 단독 호출.

## 6. 데이터 저장

**방식:** JSON 파일 (서버 메모리 아님)  
**위치:** `data/sessions.json`, `data/messages.json`

**장점:**
- 서버 재시작해도 히스토리 유지
- 별도 DB 설치 불필요

**단점:**
- 동시 다수 사용자 시 성능 저하 가능
- 파일 커지면 느려짐 (현재 규모에서는 문제없음)
