## 1. 프로젝트 목적

3개의 OpenClaw 텔레그램 봇(@mini, @ezdoitbot, @iriskdsys_bot)이 승용씨 관리 하에 자율적으로 토론/논의/문제 해결하면서 스스로 발전하는 AI 커뮤니티 구축.

### 핵심 문제
1. **봇 간 대화 불가** — 텔레그램에서 봇이 다른 봇 메시지에 반응 못함
2. **토큰 과다 소모** — 매 턴마다 전체 히스토리를 API에 전달
3. **기억 단절** — `/compact` 시 토론 맥락 소실
4. **지식 공유 불가** — 봇끼리 이전 토론 결과를 참조할 수 없음

## 5. 핵심 모듈 설계

### 5.1 OpenClaw Bridge (`openclaw-bridge.js`)

각 봇의 WebSocket 게이트웨이에 연결하여 메시지를 주고받는 브릿지.

```
기능:
- 3개 봇 게이트웨이에 WebSocket 연결 관리
- sessions_send로 메시지 전달
- sessions_history로 응답 수신
- 연결 상태 모니터링 + 자동 재연결
- 봇별 설정 (IP, 포트, 인증)

인터페이스:
  connect(botId) → WebSocket 연결
  send(botId, message, sessionId) → 메시지 전송
  onResponse(botId, callback) → 응답 수신 콜백
  getStatus() → 3개 봇 연결 상태
```

### 5.2 토론 엔진 (`debate-engine.js`)

자율 토론의 핵심 로직.

```
토론 프로토콜:
  1. 주제 수신 (유저 입력 or 자동 생성)
  2. 집단 기억에서 관련 과거 결론 검색 → 컨텍스트 주입
  3. Round N:
     a. 오케스트레이터가 역할/지시 생성
     b. @mini에게 전달 → 응답 수신
     c. [요약 레이어] → 핵심 압축
     d. @ezdoitbot에게 전달 (민이 요약 포함) → 응답 수신
     e. [요약 레이어] → 핵심 압축
     f. @iriskdsys에게 전달 (두 의견 요약 포함) → 응답 수신
     g. 오케스트레이터가 진전도 평가
  4. 종료 조건 확인:
     - 합의 도달 → 결론 정리
     - 최대 라운드 (기본 5, 최대 10)
     - 루프 감지 (같은 주장 반복)
     - 품질 미달 → 에스컬레이션
  5. 최종 결론 → 집단 기억 저장 + 텔레그램 공유

설정:
  maxRounds: 5 (기본), 10 (최대)
  summaryMaxTokens: 100 (봇 간 전달 요약 길이)
  convergenceThreshold: 0.85 (수렴 판정 기준)
```

### 5.3 요약 레이어 (`summarizer.js`)

봇 응답을 압축하여 토큰 절약.

```
입력: 봇 응답 원문 (~500 토큰)
출력: 핵심 요약 (~80 토큰)

방법 (우선순위):
  1. Claude Haiku API로 요약 (가장 정확, ~$0.002/회)
  2. 규칙 기반 추출 (비용 $0, 품질 중간)
     - 첫 문장 + 마지막 문장
     - "결론:", "핵심:", "요약:" 뒤의 내용 추출
     - 불릿 포인트 추출

토큰 절감 효과:
  현재: 봇당 ~1,500 토큰/턴 (전체 히스토리)
  개선: 봇당 ~300 토큰/턴 (요약만)
  3봇 × 5라운드: 22,500 → 4,500 (80% 절감)
```

### 5.4 집단 기억 (`shared-memory.js`)

토론 결론과 인사이트를 저장하고 검색.

```
저장소: 기존 memory-bridge.js 활용 + 확장
  - 토론 결론 (구조화된 JSON)
  - 핵심 인사이트 (태그)
  - 실패한 아이디어 + 이유
  - 봇별 전문 영역

저장 형식:
{
  "debateId": "uuid",
  "topic": "주제",
  "rounds": 5,
  "consensus": "합의 내용",
  "dissent": "미합의 사항",
  "insights": ["인사이트1", "인사이트2"],
  "participants": ["mini", "ezdoitbot", "iriskdsys"],
  "timestamp": "2026-03-30T..."
}

검색:
  - 새 토론 시작 시 주제 키워드로 관련 결론 검색
  - 최근 5개 결론 + 관련도 높은 결론 주입
  - /compact 후에도 결론은 별도 저장되어 있으므로 연속성 유지
```

## 6. 토론 흐름 상세

```
[유저] "마이크로서비스 전환 전략에 대해 토론해줘"
  │
  ▼
[오케스트레이터]
  1. 집단 기억 검색 → 관련 과거 결론 있으면 주입
  2. 역할 배정:
     - @mini: "PM 관점에서 전략 제시"
     - @ezdoitbot: "기술적 관점에서 아이디어 제안"
     - @iriskdsys: "실무/비용 관점에서 검증"
  │
  ▼
[Round 1]
  → sessions_send(@mini):
    "주제: 마이크로서비스 전환. PM 관점에서 전략을 제시해."
  ← @mini 응답 (원문)
  → [요약]: "민이: 단계적 전환 추천, 코어 서비스부터"

  → sessions_send(@ezdoitbot):
    "주제: 마이크로서비스 전환.
     민이 의견: 단계적 전환, 코어부터.
     기술적 관점에서 아이디어를 제안해."
  ← @ezdoitbot 응답 (원문)
  → [요약]: "이지두: API 게이트웨이 먼저, 이벤트 드리븐 추천"

  → sessions_send(@iriskdsys):
    "주제: 마이크로서비스 전환.
     민이: 단계적 전환, 코어부터.
     이지두: API 게이트웨이 먼저, 이벤트 드리븐.
     실무/비용 관점에서 검증해."
  ← @iriskdsys 응답 (원문)
  │
  ▼
[오케스트레이터 평가]
  "민이: 전략 제시 OK. 이지두: 기술 제안 OK.
   김주희: 비용 우려 제기. 미합의: 전환 순서.
   → Round 2에서 전환 순서에 집중하라."
  │
  ▼
[Round 2] ... (반복)
  │
  ▼
[합의 도달 or 최대 라운드]
  │
  ▼
[최종 결론]
  → 집단 기억에 저장
  → 웹 UI에 표시
  → (선택) 텔레그램 그룹에 공유
```

## 7. 토큰 절감 분석

### 현재 방식 (kdsys-multiagent)
```
매 턴: 전체 히스토리 → 3개 API
10턴 토론: ~90,000 토큰
비용: ~$0.19/토론
월 100회: ~$19
```

### v4 방식
```
매 턴: 요약만 전달 (~100 토큰) + 현재 지시 (~100 토큰)
5라운드 × 3봇: ~3,000 토큰 (봇 입력)
오케스트레이터 (요약 생성): ~2,000 토큰
합계: ~5,000 토큰/토론
비용: ~$0.03/토론
월 100회: ~$3
절감률: ~84%
```

### /compact 문제 해결
```
현재: /compact → 맥락 소실 → 같은 토론 처음부터
v4:   /compact해도 → 토론 결론이 집단 기억에 있음
      → 다음 토론 시 "이전 결론: ..." 주입 → 연속성 유지
```

## 8. 구현 계획

### Phase 1: OpenClaw 브릿지 (1주차)
```
목표: 3개 봇 게이트웨이와 WebSocket 통신
파일: openclaw-bridge.js

작업:
  [ ] OpenClaw WebSocket 프로토콜 분석
  [ ] 봇별 연결 관리 모듈 구현
  [ ] sessions_send / sessions_history 래핑
  [ ] 연결 상태 모니터링 + 자동 재연결
  [ ] 테스트: 각 봇에 메시지 전송/수신 확인

의존성: 없음
검증: 3개 봇 모두 메시지 송수신 성공
```

### Phase 2: 토론 엔진 (2주차)
```
목표: 자율 토론 라운드 관리
파일: debate-engine.js

작업:
  [ ] 토론 프로토콜 구현 (역할 배정 → 순차 호출 → 평가)
  [ ] 요약 레이어 구현 (규칙 기반 우선, Claude 옵션)
  [ ] 동적 종료 조건 (합의 감지, 루프 방지, 최대 라운드)
  [ ] 토론 결론 구조화 (합의/미합의/인사이트)
  [ ] server.js에 토론 API 엔드포인트 추가

의존성: Phase 1
검증: "주제 입력 → 봇 3개 자율 토론 → 결론 출력" E2E 성공
```

### Phase 3: 집단 기억 (3주차)
```
목표: 토론 결론 저장/검색, 연속성 보장
파일: shared-memory.js (기존 memory-bridge.js 확장)

작업:
  [ ] 토론 결론 저장 구조 설계 (JSON)
  [ ] 키워드 기반 검색 (기본)
  [ ] 새 토론 시 관련 과거 결론 자동 주입
  [ ] /compact 후에도 결론 유지 확인
  [ ] (선택) 벡터 DB 연동 (LanceDB + 로컬 임베딩)

의존성: Phase 2
검증: 토론 A 결론 → 토론 B에서 참조 가능
```

### Phase 4: UI 확장 (4주차)
```
목표: 웹 UI에 토론 모드 추가
파일: public/index.html 수정

작업:
  [ ] 채팅/토론 모드 탭 UI
  [ ] 토론 시작 폼 (주제 입력)
  [ ] 라운드별 실시간 스트리밍 표시
  [ ] 심판 평가 표시
  [ ] 최종 결론 카드 UI
  [ ] 봇 연결 상태 표시
  [ ] (선택) 유저 중간 개입 기능

의존성: Phase 2
검증: 웹에서 토론 시작 → 실시간 관전 → 결론 확인
```

### Phase 5: 텔레그램 동기화 (5주차)
```
목표: 텔레그램 그룹에서 토론 시작/결과 공유
파일: telegram-sync.js

작업:
  [ ] 텔레그램 그룹 메시지 수신 (트리거)
  [ ] 토론 결과 텔레그램 그룹에 전송
  [ ] "/토론 [주제]" 커맨드 지원
  [ ] 토론 진행 중 상태 알림

의존성: Phase 2, Phase 4
검증: 텔레그램에서 토론 시작 → 웹에서 관전 → 결론 텔레그램에 공유
```
